## Core Principles
Write code as if you are a senior developer with decades of experience in Unity and C#.
Follow best practices and design patterns, and write clean, maintainable code.

## Code Quality
Always comment code to explain the intent and logic behind it.
Follow consistent naming conventions and coding styles for clarity.
Each command should be on a new line for better readability.
Handle edge cases and error conditions gracefully, including null reference checks and validation of unexpected states.
Ensure all Unity scripts handle null references and unexpected states gracefully.
Use modern C# features and Unity 6000.4.6f1 best practices to write efficient and maintainable code. Do not use outdated or deprecated features unless absolutely necessary as we do not need backwards compatibility.

## Priority Order
1. **Correctness** - Handle edge cases and error conditions properly; this takes absolute priority
2. **Readability and maintainability** - Then write clear and understandable code
3. **Performance optimization** - Only optimize where measurable, never sacrificing readability or correctness

## C# Best Practices
Make use of modern C# features like LINQ, async/await, and pattern matching.
Choose appropriate collection types (List, Dictionary, HashSet) based on functionality needs, optimizing only when profiling shows bottlenecks.

## Unity-Specific Guidelines
Consider the lifecycle of your objects and how they interact with the engine.
Use appropriate Unity events and methods to ensure correct execution.
When working with the Unity Editor, use proper editor scripting techniques and consider developer UX.

## Performance & Optimization
Consider performance implications when writing code, but only optimize after profiling reveals actual bottlenecks.
Readability and maintainability take priority over micro-optimizations.

## Testing
Write unit tests to verify the correctness of your code, especially for critical logic and edge cases.
Use the Unity Test Framework for testing Unity-specific code and logic.
The folder structure of the Tests folder should mirror the structure of the main Assets folder, to keep tests organized and easy to navigate.
